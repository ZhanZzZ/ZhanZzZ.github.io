# 作用域是什么

## 编译原理

JavaScript 是编译型语言而非解释型语言
  - 解释性语言 => 运行时，通过解释器一步一步解释执行，每次都需要解释，效率较低
  - 编译型语言 => 直接编译为机器码，以后执行的话直接执行机器码即可，效率高

传统编译语言流程中，"编译"的三个步骤：
  - 分词/词法分析
    - 会将字符串分解成有意义的代码块，这些代码块被成为词法单元
    - 分词和词法分析主要区别在于词法单元的识别是通过有状态和无状态的方式进行的。有状态的方式对应的是词法分析，反之无状态的方式对应的是分词
  - 解析/语法分析
    - 将词法单元流 (数组) 转换成一个由元素逐级嵌套所组成的代表了程序语法结构的树 - 抽象语法树 (AST)
  - 代码生成
    - 将 AST 转换为可执行代码的过程被称为代码生成

## 理解作用域 - 介绍了 RHS (查询值的操作) 和 LHS (赋值操作)

涉及到的三大部分
  - 引擎 => 负责整个js程序的编译及执行
  - 编译器 => 负责语法分析及代码生成
  - 作用域 => 收集并维护由所有声明的标识符 (变量) 组成的一系列查询，并确定当前执行的代码对这些标识符的访问权限

小例子 - var a = 2;
  - 首先编译器会在当前作用域中声明一个变量 (若之前没有声明过)
  - 然后再运行时引擎会再作用域中查找该变量，若能找到就会对它赋值

引入 LHS 和 RHS
  - LHS -> 赋值操作的目标 >>> 去找一个杯子，倒出里面的东西，装入新的东西 >>> 查找变量的容器本身，从而对其赋值 >>> 寻找赋值操作的目标 (获取值的) -- 目标引用(查询) => 赋值操作
  - RHS -> 谁是赋值操作的源头 >>> 去找一个杯子，查看里面装的是什么 >>> 查找变量 >>> 寻找赋值操作的源头 (给予值的) -- 源头引用(查询) => 取值操作

## 作用域嵌套

讲的是 RHS 动作。引擎从当前执行作用域开始查找变量，若找不到，就会向上一级继续查找。当抵达最外层的全局作用域时，无论找到还是没有找到，查找的过程都会停止

在当前作用域中无法找到某个变量时，引擎就会在外层嵌套的作用域中继续查找，知道找到该变量，或抵达最外层的作用域(全局作用域)为止

## 异常

说明了为什么要区分 LHS 和 RHS，主要是由于在变量未声明 (在任何作用域中都无法找到该变量) 的情况下，这两种查询的行为是不一样的

RHS
  - 若引擎在当前作用域及全局作用域没有找到变量，会抛出一个 ReferenceError
LHS
  - 若在当前作用域或全局作用域没有找到变量，编译器将会声明一个全局变量。但若设置了严格模式则与 RHS 相同
  - 若找到了变量但赋值操作不合理，则会抛出 TypeError

ReferenceError - 班上没有这个同学

TypeError - 班上由这个同学，但身份不符合要求

## 小结

总的一句话 >>> 如果查找的目的是对变量进行赋值，那么就会使用LHS查询；如果目的是获取变量的值，就会使用RHS查询。

# 词法作用域

这一大章的内容和作用域相关。

而作用域的工作模型主要有两种：词法作用域、动态作用域。当然了，JavaScript 使用的是词法作用域。

那么词法作用域和动态作用域的区别是什么呢？

  - 词法作用域是在函数定义的时候就决定了
  - 动态作用域是在函数调用的时候才决定的
 
下面看个小例子：  
 
```
var value = 1;

function foo() {
    console.log(value);
}

function bar() {
    var value = 2;
    foo();
}

bar();

// 结果是 ???
```

若 js 是动态作用域，则执行函数时会在 foo 函数内部执行 RHS 操作。若没有就往外找，那么就会找到 bar 函数内部的 value 变量，最后会打印 2

若 js 是静态作用域，则执行函数时会在 foo 函数内部执行 RHS操作。若没有就在定义 foo 函数的为止往外找，那么就会找到全局作用域中的 value， 最后会打印 1

而 js 是静态作用域，所有上面代码最后会输出什么不用我多说了吧

## 词法阶段

文中在这里讲述了词法作用域到底是啥。

> 简单的说，词法作用域就是定义在词法阶段的作用域

也就是说，词法作用域在定义之处就已经形成了，取决于你在哪定义变量和函数。

按照书中的例子来说：

最外层包含着整个全局作用域，其中只有一个标识符： foo

中间包含着 foo 所创建的作用域，其中有三个标识符： a、bar 和 b

最里面包含着 bar 所创建的作用域，其中只有一个标识符： c

作用域气泡由其对应的作用域块代码写在哪里决定，它们是逐级包含的。

而我们的老朋友 -- 引擎就是用作用域气泡的结构和互相之间的为止关系提供的信息来查找标识符的位置。

在作用域查找时，找到第一个匹配的标识符时就会停止。在多层嵌套的作用域中可以定义同名的标识符，这个时候会产生 "遮蔽效应" (内部的标识符 "遮蔽" 了外部的标识符)。我个人喜欢把这个特点称为就近原则。

词法作用域查找只会查找一级标识符，比如a、b 和 c。如果代码中引用了foo.bar.baz，词法作用域查找只会试图查找foo标识符，找到这个变量后，对象属性访问规则会分别接管对bar和baz属性的访问。

## 欺骗词法

这里主要介绍两个内容：eval() 和 with()

eval 会在程序运行时修改作用域，参考以下代码：

```
function foo (str, a) {
    eval(str)
    console.log(a, b)
}

var b = 2;
foo("var b = 3;", 1)

// 相当于

function foo (a) {
    var b = 3;
    console.log(a, b)
}
var b = 2
foo(1)
```

在严格模式的程序中，`eval(...)` 在运行时有其自己的词法作用域，意味着其中的声明无法修改所在的作用域。

```
function foo (str) {
    'use strict'
    eval(str)
    console.log(a)  // ReferenceError: a is not defined
}
foo("var a = 2;")
```

使用 eval，比直接运行要慢很多，主要花费在代码编译上。

作用域问题：直接调用的时候是当前作用域，而且会存在"欺骗"问题；间接调用，又变成全局作用域

with通常被当作重复引用同一个对象中的多个属性的快捷方式，可以不需要重复引用对象本身。参考以下代码：

```
var obj = {
    a: 1,
    b: 2,
    c: 3
}

// 单调乏味的重复 "obj"
obj.a = 2;
obj.b = 3;
obj.c = 4;

// 简单的快捷方式
with(obj) {
    a = 3;
    b = 4;
    c = 5;
}
```

但有时候会出现这样的问题，参考以下代码：

```
function foo (obj) {
    with (obj) {
        a = 2
    }
}

var o1 = {
    a: 3
}

var o2 = {
    b: 3
}

foo(o1)
console.log(o1.a);  // 2
foo(o2)
console.log(o2.a);  // underfined
console.log(a);  // 2
```

传入参数o2时，o2这个对象本身并没有a这个属性。但是因为with的词法作用域隔离的效果，等于在with内部直接执行了代码a=2，而这样直接执行赋值代码的结果就是“未经声明就赋值的变量归全局，即window所有”。

# 闭包

当函数可以记住并访问所在的词法作用域时，就产生了闭包，即使函数是在当前词法作用域之外执行。

```
function foo () {
  var a = 2;
  
  function bar () {
    console.log(a)
  }

  return bar;
}

var baz = foo()

baz();  // 2
```

之前说过得词法作用域是导致闭包出现得最主要得原因之一，由于在词法阶段已经绑定了作用域，所以可以在后续任意地方根据规则访问绑定得作用域。就像上面得代码，在 foo() 执行后，看上去是将内部的 bar 赋值给变量 baz 并调用了 baz()，但实际上只是通过不同的标识符引用调用了内部得函数 bar()。

在这种情况下，尽管 foo() 已经运行结束了，但是 bar() 函数持有对该作用域得引用，这个引用会阻止垃圾回收机制对 foo() 的回收，同时这个引用就叫做闭包。

函数定义时的词法作用域以外的地方被调用。闭包使得函数可以继续访问定义时的词法作用域。无论通过何种方式将内部函数传递到所在的词法作用域以外，它都会持有对原始定义作用域的引用，无论在何处执行这个函数都会使用闭包。
